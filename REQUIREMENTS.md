## 1. 背景と目的

### 1.1 背景
公式スケジュールページ（`claude.com/code-with-claude/tokyo` / `/tokyo-extended`）は
**縦リスト＋トラックタブ切替**形式で、同一時間帯に並行する複数トラックを
横並びで見比べることができない。3トラック同時進行のイベントにおいて
「この時間、どのセッションに出るか」の意思決定を視覚的に支援できていない。

### 1.2 本ツールが解決する課題
- **並行トラックの同時可視化**: 3列タイムラインで同時刻の選択肢を一目で比較
- **競合の自動検出**: あるセッションを選ぶと、時間が重複する他セッションを淡色化＋自動で置き換え（PC）／警告＋置き換え（スマホ）
- **個人の参加計画の保存**: チェックした予定を端末ローカルに永続化

### 1.3 制作者にとっての目的
- Claude Code で作った実用ツールとしての提示（作品性）
- 「自分用に作った副産物の共有」というトーンでの X 告知
- 当日、参加者に少しでも実際に使ってもらえる実用性の確保

### 1.4 非目的（やらないこと）
- サーバー／DB連携、ユーザー間共有、リアルタイム同期
- 公式の代替・置き換え（あくまで補助・非公式）
- スケジュール変更への自動追従（静的データ）

## 2. ターゲットと利用シナリオ

### 2.1 ターゲットユーザー
Code with Claude Tokyo 2026 の参加者（開発者中心）。一部はワークショップ参加者で
ラップトップ持参。X 経由でリンクを開く層はスマホが多い。

### 2.2 利用シナリオ（優先順）
| # | シナリオ | 主デバイス | 重要機能 |
|---|---------|-----------|---------|
| S1 | イベント前夜、自宅でセッション計画を立てる | PC | 3列タイムライン・競合表示＋置き換え |
| S2 | 移動中・会場でXからリンクを開いて閲覧 | スマホ | 縦リスト・タブ切替・チェック |

| S4 | ワークショップ参加中に全体感を再確認 | PC/スマホ | 両レイアウト |

### 2.3 デバイス構成の前提
「8割スマホ」という一般イベントの想定よりラップトップ率は高いが、
**閲覧の入口（Xリンク）はスマホ優位**。よってPC・スマホ双方を一級市民として扱う。

## 5. 機能要件

### FR-1: 日付タブ切替
- ヘッダー直下に Day 1 / Day 2 のタブを配置（sticky）
- タブ切替で当日のデータ・トラック・配色・保存キーを切替えて再描画
- 各日のチェック状態は独立して保持（日をまたいでも消えない）
- 選択中のタブは URL ハッシュ（`#day1` / `#day2`）に反映し、リロード・共有で復元
- **初期表示の優先順位**（上から順に評価し、最初に合致したものを採用）:
  1. URLハッシュ `#day1` / `#day2` があればそれを優先（共有URLの再現性）
  2. アクセス日が 2026-06-11 なら Day 2
  3. アクセス日が 2026-06-10 なら Day 1
  4. それ以外（前日まで・翌日以降）は Day 1（既定）

### FR-2: PCタイムライン表示
- 3列グリッド、時間軸を左に固定、セッションを絶対配置
- **列＝ステージ**（Day1: Main/Breakout/Workshop、Day2: Founder/Builder/Workshop）
- **色＝トラック**（Day1: Research/Platform/Code を teal/purple/coral で。Day2 は列と同一）
- ステージ内は時間重複がないため**段組（stagger）は不要**。シンプルな1列1枠で描画
- Day1 の各セッションカードはトラック色で塗り、列見出しにステージ名を表示
- 全幅特別バー（朝食・基調講演・昼食・レセプション）はゴーストバンド＋ラベルバー方式
- 既存の視覚トークン（配色・タイポ・角丸）を踏襲し、見た目の一貫性を保つ
- モバイル用の追加は原則 `@media` 内に閉じ、PC表示への意図しない回帰を避ける

### FR-3: スマホ縦リスト表示（新規）
- 時間昇順で全セッションを縦に並べる
- 同一時間帯（開始時刻が重なる）のセッションは**タイムスロット見出し**でグルーピング
  - 例（Day1）: `10:30` の下に Main / Breakout / Workshop のカードを並べ、
    カード色で Research / Platform / Code（トラック）を示す
- 各セッションカード要素:
  - 時刻（開始–終了）
  - トラックカラーの左ボーダー＋トラックタグ
  - タイトル
  - スピーカー
  - JPタグ（日本語セッション）
  - ステージ表記（Day1のみ）
  - **専用チェックコントロール**（円形ボタン、44px相当のタップ領域）
  - **展開トグルのアフォーダンス**（シェブロン等。タップで説明文を開閉）
- カード本体タップ＝説明展開、チェックは専用ボタン（FR-4 / FR-9 参照）
- 全幅特別バー（昼食等）はリスト内にインラインで時系列配置（説明文は持たない）
- トラックフィルターはスマホでも維持（チップで絞り込み）

### FR-4: 参加チェックとトグル
- **チェックは専用コントロール（円形ボタン）で行う**。カード本体タップは説明展開（FR-9）に割当て、
  チェックと展開を明確に分離する（誤操作防止）
- チェックボタンの操作で参加ON/OFF。状態は即座に localStorage へ保存（日別キー）
- チェックONのセッションは視覚的に明示（アウトライン＋チェックバッジ）
- チェックボタンは色だけに依存せずアイコン（✓）でも示す
- **配置を固定**：カード右上にチェックボタン。展開シェブロンはタイトル横／右下に置き、
  両者を物理的に離して誤タップを防ぐ（§7.4）
- **タップ領域**：スマホ（タッチ）は44px相当を確保。PC（マウス）は精度が高いため
  小さめの円形ターゲットで可（タイムラインの小型カードに収める）
- 特別枠（朝食・昼食等）も従来どおりチェック可能

### FR-5: 競合検出（全デバイス担保・「淡色化＋置き換え」方式）

> 設計判断: 旧版の「ロック＝操作不可」は「チェックで置き換え」と矛盾していたため廃止。
> **競合は視覚的に示すが操作は常に可能**とし、チェックした瞬間に競合を自動解除して
> 新しい選択に置き換える。これによりPC/スマホの挙動が完全に一致し、当日の
> 「外してから選び直す」手間も発生しない。

- **共通の挙動**: あるセッションをチェックすると、時間が重複する既選択セッションを
  自動的に解除して新しい選択へ置き換える（時間は重複できない、という不変条件を保つ）
- **PC（タイムライン）**: 選択中セッションと時間が重複する未選択セッションを
  **淡色化**して「今は両立しない枠」を視覚提示する。淡色化されていても
  **チェックは可能**で、チェックすれば競合を解除して置き換わる
- **スマホ（縦リスト）**: 選択中セッションと時間が重複する他セッションに
  `⚠ 選択中のセッションと時間重複` を明示する。チェックで同様に置き換え
- 競合判定ロジック（`ov()` による時間重複計算）は PC/スマホで**共通化**する
- 状態だけに依存させず、淡色化（PC）／警告テキスト＋アイコン（スマホ）の
  双方で冗長に伝える（アクセシビリティ NFR-3 と整合）

### FR-6: 参加予定サマリー
- 選択セッションを時刻順に一覧表示（両レイアウト共通DOM）
- 件数表示、各行に時刻・トラックタグ・タイトル・JPタグ・削除ボタン
- 「すべてクリア」ボタン
- 0件時はプレースホルダー

### FR-7: トラックフィルター
- トラック凡例チップのタップで該当トラックを淡色化（絞り込み）
- **挙動**: 各トラックは独立したトグル。**複数トラックを同時に絞り込み可能**で、
  同じチップを再タップすると元に戻す（チップ自体も淡色で状態を示す）
- フィルターは表示上の絞り込みのみで、チェック状態には影響しない
- 当日のトラック構成（Day1=r/p/c、Day2=f/b/w）に追従
- PC・スマホ両レイアウトで同一に機能する

### FR-8: 状態の永続化と同期
- localStorage に日別キーで保存
- **同一日のPC用DOMとスマホ用DOMは同じ状態（`ck`セット）を共有**し、
  レイアウトが切り替わっても（リサイズ含め）チェック状態が一致する
- これが本実装の**最大の同期リスクポイント**。状態は単一の真実源（`ck`）とし、
  両レイアウトのDOMはそこから描画する

### FR-9: セッション説明の表示（媒体別に方式を確定）
- 説明文は公式ページ準拠（`desc` フィールド、§8.1）。全セッション分を保持
- **チェックとは独立**：チェックは専用ボタン（FR-4）、本体タップは説明表示のみ
- 説明の表示状態は永続化しない（一時的UI状態。リロードでリセット）
- アニメーションは控えめに。`prefers-reduced-motion` 時はアニメを無効化
- 特別枠（朝食・昼食等）は説明を持たないため対象外

**PC（タイムライン）= 中央モーダル（確定）**
- カード本体クリックで**中央モーダル／ダイアログ**を開く
- 理由: タイムラインのカードは小型・説明文は長文。ポップオーバーは画面端で切れ、
  スクロールと干渉し可読性が落ちる。モーダルはレイアウトを崩さず長文を読みやすく、
  スマホのインライン展開と役割分担できる
- モーダル内容: 時刻 / タイトル / 登壇者 / ステージ・トラック / 説明文、
  および**チェックトグル**（モーダルからも参加登録できる）
- 閉じる手段: `Esc` キー / 背景（バックドロップ）クリック / 明示的な閉じるボタン
- フォーカストラップと `aria-modal="true"` を付与（アクセシビリティ）
- 同時に開くのは1つ（モーダルなので排他）

**スマホ（縦リスト）= カード内インライン展開（確定）**
- カード本体タップでカード直下にアコーディオン展開（複数同時展開を許容）
- 展開トグルは `aria-expanded` を持ち、シェブロンの回転で開閉を示す
- 縦フロー配置のため高さ展開でレイアウトは崩れない

### FR-10: 時間軸ステータス（現在時刻ハイライト＋開催前カウントダウン）

> 設計判断（採用方式と不採用方式）:
> 「右上に常設のライブ秒読みカウントダウンを置く」案は **不採用** とする。理由は
> (1) 賞味期限が開催までの数日に限られ、終了後は無意味化する、
> (2) 左揃えヘッダー（eyebrow / タイトル / リード / メタ）に重いウィジェットを足すと
>     特にスマホで窮屈になる、
> (3) 秒刻みの演出は本ツールのミニマルな信頼感（§7.5）とレビュー指針
>     「実用7:作品3」に反する（“作品盛り”側に倒れる）。
> 代わりに、**時間軸上で「今どこか」を示す軽量な状態表示**として設計し、
> 開催前＝カウントダウン、当日＝現在時刻ライン（タイムライン本体）へと
> 主役を引き継ぐ、一本の時間軸ナビとして統合する。

**10.1 現在時刻ハイライト（当日・実装済み）**
- **表示中の日付が端末日付と一致する当日のみ**自動表示（非イベント日は一切表示しない）
- **PC**: タイムラインに現在時刻の横ライン＋時間軸に時刻ピル。進行中セッションを
  アクセント（coral）リングで強調し、終了済みセッションを淡色化（＝これから行ける枠に視線を集める）
- **スマホ**: 時系列リストに `● 現在 HH:MM` ディバイダを次の枠の直前に挿入。
  進行中カードにアクセントリング＋「進行中」表記、終了済みカードを淡色化
- 約30秒間隔で更新。`prefers-reduced-motion` を尊重（点滅・派手なアニメは持たない）
- 時刻取得は単一関数 `getNow()` に集約。`?now=YYYY-MM-DDTHH:MM` または `?now=HH:MM` で
  擬似時刻を指定でき、検証・直前デモ・スクリーンショットに用いる（本番は未指定＝実時刻）

**10.2 開催前カウントダウンチップ（追加提案）**
- **配置**: イベントメタ行（`June 10 · Tokyo · 08:00–20:00 JST`）の末尾に小さなチップとして添える。
  独立した右上ウィジェットは作らない（上記設計判断）
- **基準時刻**: 表示中の日の開始時刻（Day1=6/10 08:00 / Day2=6/11 08:30 JST）。
  タブ切替で基準日に追従する
- **状態遷移**（`getNow()` を再利用。**日単位**で算出し秒読みはしない）:

  | 状態 | 表示例 |
  |------|--------|
  | N日前（N≥2） | `開催まであと N日` |
  | 前日 | `明日開催` |
  | 当日・開始前 | `本日開催 · まもなく` |
  | 当日・開催中 | `開催中`（主役は 10.1 の現在時刻ラインに委譲） |
  | 終了後（翌日以降） | 非表示（任意で最終日翌日まで控えめな謝辞も可） |

- **トーン**: `ti-clock` ＋ニュートラルなアウトラインピル、アクセントに coral ドット
  （ファビコン／OG画像と同系）。既存トークン（`--sf` / `--bd2` / `--coral`）に準拠
- **アクセシビリティ**: 残り日数はテキストで明示し色のみに依存しない。
  更新頻度が低いため `aria-live` は不要
- **保守性**: 軽量・**自己引退型**。状態が尽きると自動で非表示になり、開催後の撤去作業を残さない
- **告知との関係**: 計画喚起・鮮度の効果が出るのは開催直前の数日。X告知のタイミングで投入する想定

## 6. 非機能要件

### NFR-1: レスポンシブ設計
- ブレークポイント: `max-width: 720px` をスマホ境界とする
- 切替方式: PC用DOMとスマホ用DOMを併存させ、`@media` で `display` 切替
  ```css
  .view-mobile { display: none; }
  @media (max-width: 720px) {
    .view-desktop { display: none; }
    .view-mobile  { display: block; }
  }
  ```
- **既存PC CSSへの影響を最小化**（媒体クエリ内のみでモバイル定義）
- 中間幅（タブレット 481–720px）はスマホ縦リストを採用
- 横向き・折りたたみ端末でも破綻しないこと

### NFR-2: パフォーマンス
- 単一HTMLファイル、外部依存は最小（アイコンフォントのみCDN）
- 初回描画 1秒以内（4G/中位スマホ想定）
- タブ切替・チェックは体感即時（< 100ms）

### NFR-3: アクセシビリティ
- タップターゲット最小 44×44px（スマホ）
- コントラスト比 WCAG AA 相当（本文 4.5:1 目安）
- チェック状態は色だけに依存しない（アイコン＋アウトラインで併示）
- キーボード操作・`aria-label` 付与（PCのチェック・削除・タブ）
- `prefers-reduced-motion` 尊重（トランジション抑制）

### NFR-4: ブラウザ・OS対応
- iOS Safari（最新2バージョン）、Android Chrome（最新2）
- デスクトップ Chrome / Safari / Edge（最新）
- `-webkit-` プレフィックス・`backdrop-filter` のフォールバック確認

### NFR-5: ダークモード
- `prefers-color-scheme` に追従（既存実装を維持・拡張）

### NFR-6: 保守性
- 2日分のデータを構造化し、トラック定義をパラメータ化
- 描画ロジックは日・レイアウトに対して可能な限り共通化
- マジックナンバー（時間軸スケール等）は定数に集約

## 7. デザイン要件（UI/UX）

### 7.1 デザインの基準
既存2ファイルのデザイントークン（カラー・タイポ・角丸・余白）を正とし、
統合後も**ビジュアルの一貫性**を保つ。Anthropic のブランドカラー
（teal #1D9E75 / purple #7F77DD / coral #D85A30）を踏襲。

### 7.2 スマホ縦リストのUX原則
- **スキャン性**: タイムスロット見出しで「今この時間に何があるか」が即わかる
- **親指操作**: チェックは大きめのタップ領域、誤タップしにくい間隔
- **競合の気づき**: 警告は色＋アイコン＋テキストで冗長化、見落とさせない
- **現在地感**: 当日表示時は「現在時刻ライン」で現在地と進行中／終了済みを示す（FR-10.1・実装済み）。
  開催前はメタ行のカウントダウンチップで時間軸上の位置を示す（FR-10.2）

### 7.3 マイクロインタラクション
- チェックON/OFFの状態遷移にさりげないトランジション（reduced-motion時は無効）
- 説明の展開／折りたたみは滑らかな高さ遷移、シェブロンの回転で開閉を示す（FR-9）
- タブ切替時のコンテンツ切替は瞬時、ちらつきを出さない
- スクロール時のsticky要素（タブ・コントロール）の挙動を破綻させない

### 7.4 カードのインタラクション設計（重要）
タップ動作の役割を明確に分離し、誤操作を防ぐ:
| 操作対象 | 動作 |
|---------|------|
| カード本体（タイトル領域）タップ | 説明の表示（PC=モーダル / スマホ=インライン展開・FR-9） |
| チェックボタン（右上・専用領域）タップ | 参加チェックのトグル（FR-4・競合置き換えFR-5） |
| 展開シェブロン（スマホ） | 説明のインライン開閉（本体タップと同義） |

チェックボタンは十分な余白で隔離し、本体タップと干渉させない。
イベントは適切に `stopPropagation` し、チェック操作で説明が開かないようにする。

### 7.5 トーン＆マナー
- 非公式ツールであることを明示する控えめな注記（フッター）
- 「公式と差異がある場合は公式を優先」の旨を記載
- 通訳・JPセッションの注記を維持
- **推奨フッター文（公式引用の明示）**:
  > Session titles and descriptions are from the official Code with Claude event pages.
  > This is an unofficial personal tool; please refer to the official schedule for the latest information.

  （タイトル・説明文は公式イベントページに基づく非公式の個人ツールである旨。
  公式説明文を全文掲載するため、引用元の明示でリスクを下げる）

## 8. データ定義

### 8.1 セッションオブジェクト
```js
// Day 1（列=ステージ / 色=トラック の2属性）
{
  id:   "m1",          // 一意ID
  col:  "main",        // 配置列 = ステージ（main/brk/ws、"all" は全幅特別枠）
  tr:   "c",           // 色 = トラック（r/p/c、特別枠は "s"）
  s:    "10:30",       // 開始 HH:MM
  e:    "11:00",       // 終了 HH:MM
  ttl:  "...",         // 公式正式タイトル（短縮しない）
  spk:  "...",         // 登壇者（公式正式名・任意）
  desc: "...",         // 公式説明文（タップで展開・FR-9、特別枠は無し）
  ja:   1              // 日本語セッション（任意）
}

// Day 2（列=色=ステージ の単一属性）
{
  id:   "f1", col:"f", tr:"f", s, e, ttl, spk, desc, ja
}
```
> 列見出しにステージ名（Main/Breakout/Workshop 等）を表示。Day1 はカードを
> トラック色（tr）で塗り、ステージ（col）で列に配置する。

### 8.2 統合後のデータ保持
```js
const DAYS = {
  day1: {
    label: "Day 1", date: "6/10", title: "Code with Claude: Tokyo",
    range: ["08:00","20:00"],
    cols:   {main:"Main stage", brk:"Breakout stage", ws:"Workshop"}, // 列=ステージ
    colors: {r:"Research", p:"Claude Platform", c:"Claude Code"},      // 色=トラック
    storageKey: "cwc_tokyo_2026_day1", data: [...]
  },
  day2: {
    label: "Day 2", date: "6/11", title: "Code with Claude: Extended Tokyo",
    range: ["08:30","18:00"],
    cols:   {f:"Founder stage", b:"Builder stage", w:"Workshop"},      // 列=色=ステージ
    colors: {f:"Founder", b:"Builder", w:"Workshop"},
    storageKey: "cwc_tokyo_2026_day2", data: [...]
  }
}
```

### 8.3 localStorage キー方針
- 統合版では意味の明確な新キーを採用: `cwc_tokyo_2026_day1` / `cwc_tokyo_2026_day2`
- **旧キーからの移行は行わない**。旧キー（`cwc_tokyo_v7` 等）は未デプロイの
  ローカル開発ファイルのみで使用され、実利用者の保存状態が存在しないため移行は不要（YAGNI）

### 8.4 セッションデータの正典方針（公式表記準拠）
- **公式ページを正典とする**。タイトル・登壇者名・時刻・トラック・ステージ表記は
  公式表記に寄せる。既存2ファイルのデータは意訳・短縮が含まれるため、そのまま採用しない
- 表示上タイトルが長い場合は**CSSで折り返し・省略（ellipsis）で対応し、データ自体は短縮しない**
- JPタグ（日本語セッション）・通訳の注記は維持する
- **データソース（受領済み）**: Day1=`10June.md`、Day2=`11June.md`（公式全文）。
  これを正典とする。ただし**生貼り付けで整形が粗い**（登壇者名が連結、改行混在等）ため、
  実装時に構造化データへ**正規化**する（登壇者の分割、肩書き除去、タイトル/説明の確定）
- **照合結果**: 既存ファイルのトラック割当（Day1）は公式フィルターと**全24件一致**で検証済み。
  一方タイトルは両日で多数が短縮されていたため、公式の正式表記へ全件修正する
- Day1 セッションは `stage`（列）と `track`（色）の2属性を持たせる。Day2 は単一属性
- **説明文（`desc`）**: 両mdの詳細セクションに全セッションの公式説明文が含まれており、
  これを `desc` に格納する（FR-9 のタップ展開で使用）。特別枠（朝食・昼食等）は説明なし

## 10. リスクと対策

| # | リスク | 影響 | 対策 |
|---|--------|------|------|
| R1 | スマホDOM追加でPC表示が崩れる | 高 | `@media`内のみでモバイル定義、既存PC CSS不変更。実装後にPC回帰確認 |
| R2 | PC/スマホでチェック状態が不一致 | 高 | 状態の単一真実源（`ck`）化、両DOMはそこから描画。リサイズ時の再描画確認 |
| R3 | 2日分統合でタブ切替時の描画バグ | 中 | データ・トラック・配色・保存キーを構造化して一括スイッチ、切替を関数化 |
| R4 | スケジュール変更でデータが陳腐化 | 中 | 非公式・公式優先の注記を明示。告知文にも記載 |
| R5 | 横スクロール/段組がスマホで崩れる | 中 | スマホは縦リスト採用で回避。中間幅も縦リストに寄せる |
| R6 | アイコンフォントCDN障害 | 低 | フォールバック（テキスト/インラインSVG）検討 |
| R7 | PCタイムラインでの説明展開がレイアウトを崩す | 中 | 絶対配置のため高さ展開不可 → **PCは中央モーダルで確定**（FR-9）。スマホはインライン展開 |
| R8 | カード本体タップとチェックの誤操作 | 中 | チェックは専用ボタン＋余白で隔離、`stopPropagation` でイベント分離（§7.4） |
