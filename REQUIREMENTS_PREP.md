## 1. 背景と目的

### 1.1 背景
- 本イベントは講演の多くが**英語**で行われる。事前に「何が話されるか」と「中心語彙」を掴んでおくと、当日の聴解効率が大きく上がる。
- 制作者は 6/10 向けに **技術英語リーディング帳（全8トピック）** を作成済み。各トピックは構造が統一されている（§1.3）。
- すでにセッションプランナーをXで告知済みで、**現に使っているユーザー・これから使うユーザーがいる**。よってアップデートは**既存利用者の体験・保存データを壊さない**ことが大前提。

### 1.2 目的
- **価値の核は「英単語帳」ではなく、参加予定に紐づく予習ナビゲーション**。最優先は「**プランナーで決めた参加予定から、対応する予習教材へ最短で飛べる**」体験で、「計画」と「予習」を一本につなぐ。
- その上で **予習を「読む」だけでなく「自分の単語リストを作って持ち歩ける」体験**へ拡張する（チェック→マイ単語→印刷/書き出し）。これらは価値順では後段のため**段階導入**する（§3）。
- 制作物としての完成度を上げつつ、**実用7：作品3**（親 §7.5 のレビュー指針）を踏襲し、過剰演出をしない。

> **価値の優先順（レビュー反映）**: ①参加予定→予習教材へ直行 ＞ ②トピック優先度が分かる ＞ ③スマホで直前に語彙確認 ＞ ④チェック語をまとめる ＞ ⑤CSV/PDF書き出し。この順が §3 の R1a/R1b/R1c の線引き根拠。

### 1.3 教材1本の既存構造（8本共通・データ化の入力）
各 `01_予習用教材/NN_TopicX ....md` は以下の節を持つ。本アップデートのデータ抽出元。

| 節 | 内容 | 学習フロー上の役割 |
|----|------|------------------|
| 冒頭パラグラフ（導入） | そのトピックで何が話されるかを平易に要約 | **① 1分で概要把握** |
| 重要語彙テーブル | `# / Term / 一般的な意味 / 今回の講演文脈での意味 / 語源・由来メモ` | **② 単語確認＋チェック** |
| English Reading Passage ＋ 日本語訳 | 中心語を**太字**にした英文と対訳 | **③ 英文→訳→出典** |
| この教材が効く講演 | `講演名 / 関係度（◎○△）/ 効く理由` | プランナー連携の正典マッピング |
| 使用ソースとリンク | 出典（公式Docs・講演書き起こし・動画URL 等） | ③の出典明示 |

### 1.4 非目的（やらないこと）
- サーバー／DB連携、アカウント、クロスデバイス同期（localStorageのみ）。
- Day 2（6/11）用の予習教材作成（未作成。仕組みだけ将来拡張可能にしておく）。
- 教材本文（語彙・英文・訳）の内容改訂。本アップデートは**配信・UI・連携**が対象で、文章は原則そのまま使う。
- 既存プランナー機能（3列タイムライン・競合検出・現在時刻ライン等）の作り直し。

## 6. 機能要件

### FR-P1: 教材トップ（トピックメニュー・3分類）
- `/prep/` に、§5 の3分類で**8トピックをカード一覧**する。
- 各カードに表示: Topic番号・タイトル・分類バッジ・**1行要約**（導入冒頭から抽出可）・**チェック済み語数 / 総語数**（例 `3 / 15 語`）。
- カードタップで該当 `topic-N.html` へ遷移（同一タブ基本）。
- ヘッダーに「マイ単語リストへ」「プランナーへ戻る（`/#day1`）」の導線。
- 分類は視覚的に明確に区切る（見出し＋区切り）。色だけに依存しない（A11y）。

### FR-P2: 教材ページ（学習フロー）
- §4.3 のテンプレートで、各教材の4ブロックを上から順に提示する。
- **① 概要**: Markdown冒頭の導入パラグラフを掲載。「約1分」など所要目安の小ラベルを添える。
- **② 重要語彙**: §FR-P3。
- **③ English Passage / 日本語訳 / 出典**:
  - 英文 → 日本語訳の順で提示（教材の太字を維持）。
  - 「使用ソースとリンク」を**出典として明示**。外部リンクは `target="_blank" rel="noopener"`。
- **この講演を聞くなら覚える単語**（教材末尾の締め・**本アップデートで刷新**）:
  - 旧「この教材が効く講演」を、**行動を促す節**に作り替える。趣旨は「**下記の講演に出るなら、このトピックの語彙（上の②）を必ず押さえよう**」。
  - **表示単位＝トピック単位**: 単語は②で一度フラットに提示済みのため再掲しない。本節は**対象講演の一覧**に徹し、②へ戻す導線（「↑ ②の重要語彙を覚えて臨もう」）を添える。
  - 対象講演を、**講演の優劣ではなく「トピックとの関連性の強弱」**で段階表示する（既存教材の関係度◎○△に対応）:
    - **◎ このトピックに一番沿っている講演**
    - **○ このトピックに内容が近い講演**
    - **△ このトピックに関連のある講演**（表示過多になる場合は省略可・親FR-11.6踏襲）
  - **文言の指針**: 「最重要／重要の講演」など**講演自体に優劣があると受け取られる語は使わない**。あくまで「このトピックにどれだけ沿うか」という関連性が伝わる表現にする（◎○△は内部の関係度キーであり、UIには上記の関連性ラベルを出す）。
  - 各講演に **(a) 日付 / (b) 講演時間 / (c) 公式タイトル / (d) ステージ** を表示する（データ・正典は §8.7）。
  - **単語自体は重要度で分けない**（語彙はフラット。先の「語彙テーブル全語をチェック可」と一貫・FR-P3）。
  - 当日スケジュール（`10June.md`）に枠が無い参考録画（SF/London等）は、日付・時間を持たないため **「参考録画」** として時刻なしで併記するか、省略する。
- フッターに「← トップ」「マイ単語へ」「プランナーへ戻る（`/#day1`）」。
- 教材ページは静的HTML。タイムラインUI等の重い部品は持ち込まない（親FR-11.5）。

### FR-P3: 単語チェック
- **チェック対象＝重要語彙テーブルの全語**（制作者選択）。各行にチェックコントロールを置く。
- チェック操作で即 localStorage 保存（§8.3）。リロード後も復元。
- チェック状態は**語彙ごとの安定ID**（`t{topic}-{nn}`・§8.2）で保持し、表示順やページが変わっても一致させる。
- 視覚表現は**色のみに依存しない**（✓アイコン＋枠線）。タップ領域はスマホ44px相当（A11y）。
- ページ上部／各トピックに「チェック済み n / 総数 N」のカウンタを表示。
- 「このトピックを全チェック / 全解除」のショートカットを任意で置く（誤操作防止に確認は不要だがUndo余地として全解除は控えめに）。

### FR-P4: マイ単語リスト（集約・書き出し）
- `/prep/my-words.html` に、**全8トピック横断でチェック済みの語**を集約表示。
- 既定の並び: **トピック番号順 → テーブル内の出現順**。各行に Topic番号・Term・一般的な意味・講演文脈での意味・語源を表示。
- 各行から**個別チェック解除**が可能（教材ページと状態を共有）。0件時はプレースホルダー＋「教材で単語を選ぶ」導線。
- **書き出し（制作者選択＝CSV と 印刷(PDF) の両方）**:
  - **CSV ダウンロード**: §8.4 のスキーマ。`UTF-8 (BOM付き)` でExcel文字化けを防ぐ。ファイル名例 `cwc-tokyo-my-words.csv`。クライアント生成（`Blob`＋`a[download]`）でサーバー不要。
  - **印刷（PDF）**: 印刷専用CSS（`@media print`）で、ナビ・チェックボックス・装飾を隠し、トピック別に整形した一覧のみを出力。ユーザーはブラウザの「PDFで保存」で取得。
- 書き出し時点の対象は**現在チェック済みの語**。エクスポートは状態を変更しない（読み取り専用操作）。

### FR-P5: プランナー連携（親FR-11の実装）
- Day 1 の該当セッションに `prep` 配列（§8.5）を付与し、以下に予習リンクを描画:
  - **説明展開部**（PC=モーダル / スマホ=インライン展開）に「予習トピック」リンク。
  - **参加予定サマリー**の各行に、対応する予習トピックを短く表示＋リンク。
- 表示例: `予習: Topic 5 Tool use / MCP / Skills / Subagents`（**絶対パス `/prep/topic-5.html`** へ）。
- **リンクは絶対パス `/prep/topic-N.html`** を用いる（プランナーは `/` 直下のため相対でも解決するが、絶対で統一して事故を避ける・§4.1）。
- リンクはチェック状態に関係なく表示するが、UX上は「**予定を決めた後にサマリーから教材へ行ける**」導線を主役にする（親R9：既存利用者を混乱させない）。
- リンクは通常の `<a>`・同一タブ遷移を基本。新規タブ強制はしない。
- マッピングの正典は各教材の「この教材が効く講演」テーブル。短縮・総称は `index.html` の正式タイトルへ手動対応（親FR-11.6）。
- **R1a の核**: 本連携は R1a の必達。プランナー改変は**最小diff**（`session.prep` 付与＋リンク描画）に限定する（NFR-P1）。

### FR-P6: レスポンシブ対応（スマホ）— **本アップデートで重視**
- 教材トップ・教材ページ・マイ単語のすべてを**スマホ閲覧前提**で設計する（プランナー同様、PC/スマホ双方を一級市民に）。

## 7. 非機能要件

### NFR-P1: 既存資産との互換性（最重要）
- **公開トップURL `/` を変えない**。プランナーは従来どおり `index.html` が入口。
- 既存の **localStorageキー（`cwc_tokyo_2026_day1` / `cwc_tokyo_2026_day2`）を読まない・書かない・消さない**。本アップデートは**別名前空間の新キー**のみ追加（§8.3）。
- プランナー側の改変は**連携リンク描画の追加に限定**し、3列タイムライン・競合検出・現在時刻ライン等の既存挙動に回帰を出さない。
- React等への再構築をしない（親FR-11設計判断・R11）。

### NFR-P2: レスポンシブ設計
- 親NFR-1に準拠（`max-width:720px`境界、`@media`内にモバイル定義を閉じる）。教材ページは新規のため最初からモバイルファースト寄りに作ってよいが、**プランナー既存CSSに影響を与えない**スコープ分離（クラス接頭辞 `prep-` 等）。

### NFR-P3: パフォーマンス
- 教材ページは静的・外部依存最小（アイコンフォントのみCDN許容、無くても破綻しないフォールバック）。
- 初回描画1秒以内（4G/中位スマホ）。チェック操作は体感即時（<100ms）。
- 語彙データはページ内同梱（または `prep.js` 1ファイル）で追加リクエストを抑える。

### NFR-P4: アクセシビリティ
- タップターゲット最小44×44px、コントラストWCAG AA相当、状態は色＋アイコンで冗長化。
- チェック・書き出し・ナビにキーボード操作と `aria-label`。`prefers-reduced-motion` 尊重。
- 出典リンクはリンクテキストで行き先が分かる文言にする。

### NFR-P5: ダークモード
- `prefers-color-scheme` 追従（プランナーの既存トークンを流用）。教材ページ・印刷出力でも可読性を確保（印刷はライト基調に正規化）。

### NFR-P6: ブラウザ・OS対応
- iOS Safari（最新2）/ Android Chrome（最新2）/ デスクトップ Chrome・Safari・Edge（最新）。
- CSV ダウンロード（`Blob`/`download`）と印刷の双方を主要ブラウザで確認。

### NFR-P7: 保守性
- 教材データ（`PREP_TOPICS` / `PREP_VOCAB`）と描画ロジックを分離。8トピックを同一テンプレートで生成。`PREP_TOPICS[n].source` を生成入力にする（§8.1）。
- 語彙IDは §8.2 の方式に従う。**R1a は方式A（行順ベース暫定ID）でも可だが、R1b（チェック）では方式B（term slug）を推奨**（行順変更に強い）。方式Aを使う間は「暫定IDであり行順変更でずれる」ことを前提に運用する。

### NFR-P8: 保存の限界・前提（localStorageのみ／明示する）
> DB非依存で要件は満たせるが、保存の限界をユーザーにもドキュメントにも**明示**する（誤解防止）。
- **端末・ブラウザを変えると引き継がれない**（クロスデバイス同期なし）。
- **ブラウザのデータ削除・サイトデータ消去で消える**。
- **Safari等のプライベートブラウズ**では保存が制限／不安定な場合がある（書き込み失敗を握りつぶし、致命的エラーにしない）。
- **CSV/PDF はユーザー操作による手動保存**であり、自動バックアップではない。
- 上記をフッターまたはマイ単語ページに**短い注記**として掲示する（例: 「チェックはこの端末のブラウザにのみ保存されます」）。

## 8. データ定義

### 8.1 トピックメタ `PREP_TOPICS`
> **重要（番号不一致）**: 公開URLの Topic 番号と、原本Markdownの `NN_` ファイル番号は**一致しない**
> （例: Topic 2 の原本は `07_Topic2 ...md`、Topic 3 は `02_Topic3 ...md`）。誤対応を防ぐため、
> 各トピックに **`source`（原本md実ファイルパス）を必須**で持たせ、生成はこの `source` を入力にする。

const PREP_TOPICS = {

### 8.2 語彙データ `PREP_VOCAB` と語彙ID
> 語彙IDは R1b（単語チェック）で localStorage の保存キーになるため、**ID方式の安定性が保存の整合に直結**する。
> 2方式を定義し、**R1b 実装時は term slug を推奨**する。

**方式A（暫定・R1a/簡易）＝ 行順ベース暫定ID `t{topic}-{nn}`**
- `nn` は重要語彙テーブルの `#` 列を2桁ゼロ埋め。例: Topic5 の3番目 = `t5-03`。
- **「安定ID」ではなく行順ベースの暫定ID**。原本Markdownのテーブル**行順を変えると別IDになり、既存チェックがずれる**。短期運用（イベント前後の数日）では許容するが、原本変更時は再生成し整合に注意。

**方式B（推奨・R1bで採用）＝ term slug `t{topic}-{slug}`**
- 例: `t5-scaffolding` / `t5-tool-use`。**行順変更に強い**（並べ替えてもIDが不変）。
- slug 生成規則: 英小文字化 → 英数以外（空白・`/`・`()`・記号）をハイフンに → 連続ハイフン圧縮・前後ハイフン除去。
  - 例: `subagent / multi-agent orchestration` → `subagent-multi-agent-orchestration`、`MCP (Model Context Protocol)` → `mcp-model-context-protocol`、`eval / evaluation` → `eval-evaluation`。
- **同一トピック内で slug が衝突**した場合は `-2`,`-3` を付す（生成時に検出してログ）。slug は生成後 `PREP_VOCAB` に**確定値として焼き込む**（以後の自動再生成で揺れないよう固定）。

```js
const PREP_VOCAB = {
  5: [
    { id:"t5-scaffolding", term:"scaffolding", general:"足場、骨組み", talk:"昔は自分で組んだ周辺コード…", root:"scaffold（建築の足場）…" },
    { id:"t5-tool-use",    term:"tool use",    general:"道具の使用",   talk:"モデルが外部機能を呼び出す力…", root:"—" }
    // …テーブル全行（方式B＝slug。方式A採用時は id:"t5-01" 等）
  ]
  // 他トピックも同様
};
```

### 8.3 localStorage キー方針（**新規・既存と非衝突**）
- 追加キーは予習名前空間に限定する。**既存の day1/day2 キーには一切触れない**。
```js
// チェック済み語彙IDの配列（全トピック横断）。IDは §8.2（推奨＝term slug）
localStorage["cwc_tokyo_2026_prep_words"] = JSON.stringify(["t1-codebase","t5-scaffolding","t5-mcp"]);
```
- 命名は既存規約（`cwc_tokyo_2026_*`）に合わせ、意味が明確な接尾辞 `prep_words` を用いる。**既存 day1/day2 キーと衝突しない**。
- 旧キーからの移行は行わない（移行対象が存在しない＝YAGNI。親§8.3と同方針）。
- 破損時フォールバック: パース失敗時は空集合として扱い、既存値を破壊しない（try/catch）。
- 未知ID（原本変更等で `PREP_VOCAB` に存在しないID）は**表示時に無視**し、保存配列からは消さない（再出現時に復活させ、誤消去を避ける）。

### 8.4 マイ単語 CSV スキーマ
- 文字コード: **UTF-8（BOM付き）**。区切り: カンマ。フィールドはダブルクオートで囲み、内部の `"` は `""` にエスケープ。
- ヘッダー行 ＋ データ行（チェック済み語のみ）:

| 列 | 内容 |
|----|------|
| `topic_no` | トピック番号 |
| `topic_title` | トピックタイトル |
| `term` | 用語（英語） |
| `general_meaning` | 一般的な意味 |
| `talk_meaning` | 今回の講演文脈での意味 |
| `etymology` | 語源・由来メモ |

### 8.5 セッション側の予習参照 `session.prep`（親FR-11.3）
```js
// index.html の Day1 セッションに付与
{ id:"...", ttl:"The expanding toolkit", prep:[{ topic:5, rel:"◎" }] }
```
- 1セッションに複数教材が効く場合は複数件可。`rel`（◎○△）は**関連性の強い順**に表示（講演の優劣ではない・FR-P2）。表示文言は `PREP_TOPICS[topic].title` を参照、リンク先は `PREP_TOPICS[topic].href`（絶対 `/prep/topic-N.html`）。

### 8.6 正典・生成方針
- **教材本文の正典は `01_予習用教材/*.md`**。公開HTMLはここから生成（手作業整形でも軽量スクリプトでも可）。生成入力は §8.1 の `source`。
- 本文とプランナー側マッピングが食い違う場合は**教材本文（「この教材が効く講演」表）を正**として `prep` を更新（親§8.5）。
- 公開導線では日本語ファイル名のMarkdownへ直接リンクしない（安定英数字URL）。
- **Markdown 構造ゆれの吸収（重要）**: 8本は**完全同一構造ではない**。生成・パースはこの差を吸収すること:
  - **見出しレベルのゆれ**: 「重要語彙 / English Reading Passage」等が Topic により `##` と `###` で混在（例: Topic 4 は `###`）。**見出しレベルに依存せず、見出しテキストで節を特定**する。
  - **余分な節・前置き**: Topic 8 は本文前に `## 導入` を持ち、末尾に `## 付録…`（Topic横断の録画リンク表）がある。**`付録` は教材本文に取り込まない**。
  - **見出し名の表記ゆれ**: 「重要語彙」「重要語彙 11語」「重要語彙（8語）」等の語数付きや、語彙表前の注記行（Topic 5 の「※…太字…」）がある。**前方一致／正規化で吸収**し、注記は語彙データに混ぜない。
  - **語彙表の列ゆれ**: 列は基本 `# / Term / 一般的な意味 / 今回の講演文脈での意味 / 語源・由来メモ` だが、欠落・順序差に備え**ヘッダ名で列を取得**する。
  - **「効く講演」表の関係度ゆれ**: `◎ / ○ / △` のほか `△〜○` のような中間表記がありうる（Topic 3）。**先頭記号で正規化**（`△〜○`→○扱い等、迷えば弱い方）し、UIラベルは §8.7 の `REL_LABELS`。
  - 各 source につき**生成後に人手で1度目視確認**する（自動抽出を過信しない）。

### 8.7 対象講演データ（教材末尾「この講演を聞くなら覚える単語」用）
- 各トピックに、対応する当日講演を**関連性キー付き**で持たせる。
- **正典**: (a) 講演の**日付・講演時間・公式タイトル・ステージ**＝`10June.md`（当日スケジュール）/ (b) **関連性**＝各教材「この教材が効く講演」表の◎○△。
- UIには関連性ラベル（下記 `REL_LABELS`）を表示し、内部キー `rel`（◎○△）は表示しない。
```js
const REL_LABELS = {
  "◎": "このトピックに一番沿っている講演",
  "○": "このトピックに内容が近い講演",
  "△": "このトピックに関連のある講演"   // 表示過多になる場合は省略可
};

PREP_TOPICS[5].talks = [
  { rel:"◎", title:"The expanding toolkit",                        date:"6/10", time:"…", stage:"Main stage" },
  { rel:"◎", title:"Beyond the basics with Claude Code",           date:"6/10", time:"…", stage:"Breakout stage" },
  { rel:"○", title:"Build a production-ready agent with Claude Managed Agents", date:"6/10", time:"…", stage:"Workshop" }
  // 当日スケジュールに枠が無い参考録画（SF/London等）は date:null, time:null とし「参考録画」表示 or 省略
];
```
- 講演時刻は `10June.md` を正典に**実装時に確定**する（上記は構造例）。教材表記が短縮・総称の場合は公式タイトルへ手動対応（親FR-11.6）。
- 並び順は **◎ → ○ →（△）** の関連性順。同一関連性内は時刻昇順を基本にする。

## 9. 開発プロセス・データ互換

> 制作者の重視点（①ブランチ→マージ運用、②既存ローカル保存データを壊さない、③デプロイ先URL不変）への対応。

- **ブランチ戦略**: `main` から作業ブランチ（例 `feature/prep-materials`）を切って開発。**R1a の受け入れ基準（§12.1）を満たした段階で `main` へマージ**（R1b/R1c は追って同ブランチまたは後続ブランチで積む）。
  - 既存利用者に影響する変更（プランナー側 `index.html`）は**最小diff**に保ち、連携リンク追加に限定する。
  - レビュー後にマージ → Vercel本番反映、の順。マージ前にプレビューデプロイで実機確認できると望ましい。
- **データ互換の担保**:
  - 既存 day1/day2 キーのスキーマ・キー名を変更しない（読み書きしない新キーのみ追加）。
  - プランナーをアップデート後に開いた既存ユーザーの**チェック済み予定がそのまま復元**されることを回帰確認（§11）。
- **URL不変**: 公開トップ `/`・ハッシュ `#day1/#day2` を維持。教材は配下 `/prep/...` に増設するのみ。
